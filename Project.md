# Pokedex - Clean Architecture 과제

# 1️⃣ Intro

이 과제의 목표는 **설계**입니다.

단순히 동작하는 앱을 만드는 것이 아니라, **왜 이렇게 설계했는지** 스스로 판단하고 그 근거를 명확히 설명할 수 있어야 합니다.

Clean Architecture와 SOLID 원칙을 학습하고, 직접 적용하면서 **설계의 가치**를 경험해보세요.

**포켓몬 도감 앱 만들기**

> 포켓몬 API를 활용한 도감 앱을 Clean Architecture 기반으로 설계하고 구현합니다.

---

## 2️⃣ 학습 목표

- Clean Architecture의 핵심 원리와 의존성 규칙 이해
- SOLID 원칙을 실제 코드에 적용하고, 그 이유를 설명할 수 있는 능력
- 테스트 가능한 설계가 왜 중요한지 체감
- 설계 의도를 문서로 명확히 표현하는 능력

---

## 3️⃣ 기술 스택 및 제약사항

| 구분 | 요구사항 |
| --- | --- |

## 4️⃣ 기능 요구사항

### 메인 화면

- 포켓몬 목록을 그리드 형태로 표시 (한 줄에 3개)
- 세로 스크롤 지원
- 셀 터치 시 상세 화면으로 이동

### 상세 화면

- 포켓몬 번호, 이름(한글), 타입, 키, 몸무게 표시

### [PokeAPI](https://pokeapi.co/docs/v2)

- 목록: `https://pokeapi.co/api/v2/pokemon?limit={limit}&offset={offset}`
- 상세: `https://pokeapi.co/api/v2/pokemon/{id}/`
- 이미지: `https://raw.githubusercontent.com/PokeAPI/sprites/master/sprites/pokemon/other/official-artwork/{id}.png`

## 5️⃣ 설계 요구사항

### 커밋 헤더 `[Modify]`

(이후에 검토해보면 좋을 내용들을 기록하는 용도입니다.)

개발 과정에서 다음 상황이 발생하면 해당 커밋에 `[Modify]`헤더를 붙여주세요. 

- 코드를 추가하는 것이 아니라, 기존 코드를 수정하거나 삭제하는 경우
- 하나의 파일 수정으로 인해 다른 파일들의 연쇄 수정이 필요한 경우
- 인터페이스(Protocol)를 수정하는 경우

### 필수 구조

프로젝트는 최소 다음 레이어로 분리되어야 합니다:

- **Presentation Layer**
- **Domain Layer**
- **Data Layer**

각 레이어의 책임과 경계는 스스로 정의하세요.

### 문서 정리

(모든 걸 할 필요는 없습니다. 중요하다고 판단되는 문서를 우선순위로 해주세요.)

**1. 설계 문서**

- 아키텍처 구조도
- 각 레이어/모듈의 책임 정의
- 의존성 방향과 그 이유
- 주요 설계 결정 사항과 근거

**2. 컴포넌트 의도 설명**

- 주요 컴포넌트가 왜 이런 형태로 설계되었는지
- SOLID 원칙이 어디에 어떻게 적용되었는지

**3. 테스트 설계 문서**

- 어떤 컴포넌트를 테스트할 것인지, 왜 그 컴포넌트인지
- Mock 객체 설계 전략

**4. 단위 테스트 코드**

- 주요 컴포넌트에 대한 테스트 구현 (중요하게 생각한 컴포넌트에 대해서만, 모든 컴포넌트 X)
